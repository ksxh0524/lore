## D2: 为什么选 SQLite 而不是 PostgreSQL？

### 核心理由

Lore 是一个 **local-first** 工具——`npm install -g lore && lore`，打开即用。

| 对比维度 | SQLite | PostgreSQL |
|---------|--------|-----------|
| 安装配置 | 零配置，单文件 | 需要安装配置运行服务 |
| 数据归属 | 数据文件在用户机器上 | 需要远程服务器 |
| 向量检索 | vec0 扩展支持 | pgvector 扩展支持 |
| ORM 支持 | Drizzle ORM 优秀支持 | 同样支持 |

### 决策

SQLite + Drizzle ORM + vec0。未来多人世界可平滑迁移 PostgreSQL。

## D4: 所有 Agent 都由 LLM 驱动

### 核心决策

**不存在纯规则引擎驱动的 Agent。** 每个角色都由 LLM 驱动思考，有自己的完整人生。

### 为什么？

Lore 的核心卖点是"活的世界"。如果背景 Agent 用规则引擎硬编码行为，用户接触到的就是虚假的 NPC——没有自己的思考，没有真实的人生轨迹。

- 送外卖的也可能创业成功
- 你的同事可能在背后说你坏话
- 你暗恋的人可能主动来撩你

这些只能通过 LLM 实现，规则引擎做不到。

### 成本控制

成本控制不在 LLM 价格，在调用频率和请求队列管理：

- 频率分级：核心 Agent 每 tick，远处 Agent 每天一次
- 请求队列：LLMScheduler 统一排队，超载时低优先级延后或丢弃
- 模型分层：核心用好模型，远处用便宜模型

## D5: Agent 懒加载创建

### 决策

不是所有 Agent 在世界初始化时都创建。Agent 的创建是渐进式的：

- 初始化时只创建用户身边的核心人物
- 用户探索新区域时按需创建
- 系统 可以预判创建

### 为什么？

- 避免初始化时需要大量 LLM 调用
- 节省资源和成本
- 支持最终上万 Agent 的规模

### 冷启动问题

懒加载创建的 Agent 需要生成"过去的记忆"。方案：创建时用 LLM 生成记忆回溯。

## D6: 为什么预设用 YAML？

| 对比 | YAML | JSON |
|------|------|------|
| 可读性 | 高 | 一般 |
| 注释 | 支持 | 不支持 |
| 多行字符串 | 支持 | 需转义 |
| 编程门槛 | 低 | 需理解语法 |

预设作者可能不是程序员，YAML 更友好。

---

## D7: 上帝模式为什么是观察为主？

### 设计理念

上帝模式不是"作弊模式"，是"观察者模式"。

- 核心体验是看到所有 Agent 的思考过程——这本身就是很大的乐趣
- 可以触发世界级事件（地震等），但不能直接操控 Agent
- 间接影响比直接控制更有趣

## D9: 虚拟平台系统

### 为什么做？

平台是世界内部的"基础设施"，不是简单的社交动态流：

- Agent 刷 TikTok、发 YouTube 视频是世界真实性的体现
- 用户可以发内容看 Agent 反响——这是一种独特的互动方式
- Agent 甚至可以创业创建新平台
- 平台生态让世界更"活"

## 技术栈总览

```
LLM 层：    Vercel AI SDK + OpenAI SDK + 图片生成 API
Agent 层：   自研 Agent Runtime（借鉴 Letta/ElizaOS）
世界引擎：   自研（World Agent + TickScheduler + EventEngine）
初始化：     自研 InitAgent（LLM 生成世界和角色）
前端：      React + Vite + shadcn/ui + zustand
后端：      Fastify + WebSocket
数据库：    SQLite + Drizzle ORM + vec0
预设格式：   YAML + Zod 校验
```

核心原则：

- **通用问题用成熟工具**（LLM 调用、UI 组件、ORM）
- **差异化问题自研**（世界引擎、Agent 行为、初始化系统、虚拟平台）
- **借鉴不绑定**
- **所有 Agent 都由 LLM 驱动，行为不受限制**
